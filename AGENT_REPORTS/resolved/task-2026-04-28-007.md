# Task 007 — Fix test selectors + increase Vercel pipeline wait
Date: 2026-04-28
Status: PENDING
Source: ux-findings-2026-04-28.md (Test selector failures; pipeline timing analysis)
Branch: main

## Context

Three known test mechanics issues are causing false timeouts and misleading
the UX Agent. They were first identified in the 2026-04-28 triage session and
logged in STATUS.md. They are NOT application bugs — the app code is correct.

Additionally, the `.github/workflows/ux-agent.yml` pipeline waits only 90
seconds for Vercel to finish deploying before running Playwright. Vercel builds
for this project typically take 90–150 seconds. The 90 second wait caused the
2026-04-28 run to test against the pre-fix deployment, producing 7 false
re-confirmations of already-fixed vulnerabilities (V7, V8, V9, V11, V15, P1,
and partially V10).

## Tasks

### A — Increase Vercel deployment wait (`.github/workflows/ux-agent.yml`)

Change the `sleep` duration in the "Wait for Vercel deployment" step from
`90` to `180`:

```yaml
# BEFORE:
echo "Waiting 90s for Vercel to build..."
sleep 90

# AFTER:
echo "Waiting 180s for Vercel to build..."
sleep 180
```

### B — Fix V3/P3 waypoint Save button (waits for name input)

The Save button in WaypointSheet is disabled until the user types a name.
Tests were tapping Save immediately after the sheet opened, before filling the
name field, so the click had no effect and the test timed out.

In `tests/ux/guest.spec.js` and `tests/ux/pro.spec.js`, wherever the
WaypointSheet Save flow is exercised:

1. Wait for the name input to be visible.
2. Fill the name field with a test value (e.g. `'Test Waypoint'`).
3. Wait for the Save button to be enabled (`not toBeDisabled()`).
4. Then click Save.

Pattern (add after the sheet-open action):

```js
const nameInput = page.locator('input[placeholder*="name" i], input[placeholder*="waypoint" i]').first()
if (await nameInput.isVisible().catch(() => false)) {
  await nameInput.fill('Test Waypoint')
  await page.waitForTimeout(300)
}
const saveBtn = page.getByRole('button', { name: /save/i }).first()
await saveBtn.waitFor({ state: 'visible', timeout: 5000 })
await expect(saveBtn).not.toBeDisabled()
await saveBtn.click()
```

### C — Fix V4 track Save button blocked by BottomNav SVG

When TrackOverlay shows the Save/Discard buttons, the BottomNav SVG icons
can intercept the Save click. In `tests/ux/pro.spec.js` for the V4 test:

Replace a plain `.click()` on the Save button with `{ force: true }`:

```js
await saveBtn.click({ force: true })
```

Or scroll the button into view first:

```js
await saveBtn.scrollIntoViewIfNeeded()
await saveBtn.click()
```

Use `force: true` as it is more reliable in CI.

### D — Fix V2/V10 offline reload (use `page.goto` not `page.reload`)

`page.reload()` while the browser is offline throws
`net::ERR_INTERNET_DISCONNECTED` because the HTML shell is not cached by the
Service Worker in the Playwright test environment (SW is registered but the
test page context may not have primed the SW cache).

In all offline reload tests (`pro.spec.js` V2 and V10 tests):

Replace:
```js
await page.reload()
await waitForAppReady(page)
```

With:
```js
await page.goto(process.env.BASE_URL || 'http://localhost:5173')
await waitForAppReady(page)
```

`page.goto` with the same origin URL triggers a new navigation that the SW
*can* serve from cache (the SW intercepts `fetch` events for navigation
requests). `page.reload()` bypasses the SW in some Chromium versions.
