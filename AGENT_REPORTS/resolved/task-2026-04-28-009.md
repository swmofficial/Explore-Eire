# Task 009 — Fix test evidence: V1/V11/V15/P1 assertions use wrong metrics

## Context

Four tests are producing misleading signals because they check the wrong thing
or are structured as `expect(true).toBe(true)` (always-pass). The UX Agent
reads these as vulnerability confirmations, but they prove nothing about
whether the underlying fixes work:

### V1 — GPS track lost on reload

The test asserts `expect(true).toBe(true)` and sets annotation
`track-survived-reload: no (V1 confirmed)` because it checks for
TrackOverlay UI text ("stop tracking", "distance", "elevation").

**Problem**: TrackOverlay is only visible when `isTracking = true`. After
page reload, `isTracking` defaults to `false` (not persisted — intentional,
per task-006 constraints). The TrackOverlay hides. The test therefore
annotates "V1 confirmed" regardless of whether the trail data survived.

**task-006** (commit 2c70af7) persisted `sessionTrail` to `ee_session_trail`
in localStorage. The trail DATA survives reload; the tracking UI does not
auto-resume. The test measures the wrong thing — it should check the data, not
the UI.

### V11 — Guest waypoints lost on reload

The test asserts `expect(true).toBe(true)`. The comment reads:
> "sessionWaypoints resets on reload by definition (mapStore has no persist
> middleware)"

**Problem**: This comment is STALE. `task-002` (commit ca5445a) added the
`ee_guest_waypoints` IIFE + write pattern. The test cannot confirm or deny
whether waypoints survive reload.

### V15 — activeModule resets on reload

The test asserts `expect(true).toBe(true)`. Evidence is screenshot-only.

**Problem**: The screenshot pair shows the map after each load; the UX Agent
infers activeModule from visual context. `task-001` added `ee-module-prefs`
persist for `activeModule`. The test cannot confirm or deny this.

### P1 — Pro user sees PRO badges

The test reads badge count IMMEDIATELY after opening LayerPanel (only 500ms
wait). The Supabase profile fetch for `is_pro` is async. If the fetch hasn't
completed, `isPro = false` and badges show even though the fix is correct.

**Problem**: Race condition between async Supabase auth and badge count read.
The code fix (`{layer.pro && !isPro && <ProBadge />}` at LayerPanel:114) is
correct. The test timeout may be caused by this or by `toggles.first().click()`
waiting on a disabled toggle.

## Task

Update `tests/ux/guest.spec.js` and `tests/ux/pro.spec.js` to use direct
localStorage checks and proper auth-ready waits. No application code changed.

### A — Fix V1 (pro.spec.js)

After the reload in the V1 test, replace the TrackOverlay text check with a
direct `localStorage` read:

```js
// REPLACE the annotation block at the end of the V1 test:

  type: 'track-survived-reload',
  description: stillTracking ? 'YES (unexpected — investigate)' : 'no (V1 confirmed)',

// NEW:

const trail = (() => { try { return JSON.parse(trailJson || '[]') } catch { return [] } })();
test.info().annotations.push({
  type: 'track-survived-reload',
  description: trail.length > 0
    ? `YES — ${trail.length} points in ee_session_trail (V1 fixed)`
    : 'no — ee_session_trail empty or missing (V1 confirmed)',
});
// Now a real assertion: if we accumulated ≥1 GPS point before the reload
// (setGeolocation was called twice), the trail must survive.
// If trail is empty, the task-006 fix is not working.
if (trail.length > 0 || trailJson !== null) {
  // Data survived — V1 is fixed for this path.
  expect(trail.length).toBeGreaterThan(0);
} else {
  // Could not proof V1 — tracking may not have started (GPS not granted
  // or start button not found). Test still passes as evidence-only.
  expect(true).toBe(true);
}
```

### D — Fix P1 (pro.spec.js)

Add an auth-ready wait before the LayerPanel badge count check:

```js
// ADD before the #tour-layers-btn click in the P1 test:

// Wait for the Supabase profile fetch to complete. isPro is set async
// (useAuth → profiles.is_pro fetch → setIsPro). Without this wait, the
```
