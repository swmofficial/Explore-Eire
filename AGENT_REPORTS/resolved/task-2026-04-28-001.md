# Task 001 — Persist user preference state across reloads (V7, V8, V9, V15)

Date: 2026-04-28
Status: PENDING
Source: ux-findings-2026-04-28.md
Branch: main

## Context

Four confirmed vulnerabilities share the same root cause: key Zustand store
fields are in-memory only and reset to hardcoded defaults on every page reload.

Test evidence (2026-04-28 report):
- V7: `free V7` annotation `{"flipped":true,"tFlipped":"light","tReloaded":"dark"}` — theme reverts.
- V7: `guest V7` annotation `{"theme-initial":"dark","theme-after-flip":"light","theme-after-reload":"dark"}` — confirmed.
- V15: `guest V15` — active module reset journey completed.
- V8/V9: architectural certainty from STATE_MAP.md (tests timed out, not false negatives).

Affected stores and fields:
- `src/store/userStore.js` → `theme` (default: 'dark')
- `src/store/mapStore.js` → `basemap` (default: 'satellite'), `layerVisibility` (default: { stream_sediment: true })
- `src/store/moduleStore.js` → `activeModule` (default: 'prospecting')

All three stores currently use bare `create` from zustand with no persistence.
`zustand/middleware` is already available in the zustand package — no new dependency needed.

localStorage key convention used elsewhere in the project: `ee_onboarded`,
`ee_progress` — follow the `ee_` prefix pattern.

## Task

Add `localStorage` persistence to the three stores using Zustand's `persist`
middleware with `partialize` to persist only the specific fields listed above.
Volatile/session-only fields (sessionWaypoints, sessionTrail, toasts,
mapInstance, etc.) must NOT be persisted.

### 1. userStore.js — persist `theme`

```js
import { create } from 'zustand';
import { persist } from 'zustand/middleware';

const useUserStore = create(
  persist(
    (set, get) => ({
      // ... existing store body unchanged ...
    }),
    {
      name: 'ee-user-prefs',
      partialize: (state) => ({ theme: state.theme }),
    },
  ),
);
```

## Constraints

- BUGS.md rule 12 and 31: persisting `basemap` only affects the initial value on
  load — it does NOT change the setStyle/addDataLayers/terrain ordering on switch.
  Do not touch Map.jsx basemap switch logic.
- BUGS.md rule 24: `layerVisibility` persistence must not interfere with
  `activeMineralCategory` — that field is volatile and must NOT be persisted.
- `partialize` must exclude: `sessionWaypoints`, `sessionTrail`, `mapInstance`,
  `toasts`, `basemapPickerOpen`, `dataSheetState`, any `Open`/`Show` boolean UI flags.
- Do not change any store action signatures — only wrap the `create` call.
- `zustand/middleware` is part of the zustand package — no new `package.json` entry.
