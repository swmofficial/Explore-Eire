# Task 006 — Fix V1: Auto-persist sessionTrail to localStorage during active tracking
Date: 2026-04-28
Status: PENDING
Source: ux-findings-2026-04-28.md (Finding 7 — V1)
Branch: main

## Context

`mapStore.sessionTrail` accumulates GPS points (array of `{lat,lng,alt,heading,ts}`)
during an active tracking session. It is pure in-memory Zustand state — not
included in the persist middleware partialize — so the entire trail is lost if
the browser tab is closed, the device reboots, or the app crashes before the
user taps Save.

The UX Agent confirmed V1 with annotation `track-survived-reload: no (V1
confirmed)` in the 2026-04-28 run.

The pattern for per-key manual localStorage persistence was already established
for `sessionWaypoints` (task-002, commit ca5445a). `sessionTrail` should follow
exactly the same pattern:

- An IIFE in the store initialiser reads from localStorage on store creation.
- Each `appendSessionTrailPoint` call also writes the updated array to
  localStorage.
- `clearSessionTrail` also removes the localStorage key.

This covers crash/tab-close recovery. It does NOT cover mid-tracking reload
UI (e.g., prompting the user to resume a recovered track). That UX is out of
scope; the data survives and the TrackOverlay will resume correctly because
`sessionTrail` will be non-empty and `isTracking` will need to be handled
separately (see Constraints).

## Task

Modify `src/store/mapStore.js` only. Three changes, all within the store body:

**1. IIFE hydration on init** (replace the sessionTrail initial value):

```js
// BEFORE:
sessionTrail: [],

// AFTER:
sessionTrail: (() => {
  try {
    return JSON.parse(localStorage.getItem('ee_session_trail') || '[]')
  } catch { return [] }
})(),
```

**2. Write to localStorage in appendSessionTrailPoint**:

```js
// BEFORE:
appendSessionTrailPoint: (pt) =>
  set((state) => ({ sessionTrail: [...state.sessionTrail, pt] })),

// AFTER:
appendSessionTrailPoint: (pt) =>
  set((state) => {
    const next = [...state.sessionTrail, pt]
    try { localStorage.setItem('ee_session_trail', JSON.stringify(next)) } catch {}
    return { sessionTrail: next }
  }),
```

**3. Remove localStorage key in clearSessionTrail**:

```js
// BEFORE:
clearSessionTrail: () => set({ sessionTrail: [] }),

// AFTER:
clearSessionTrail: () => {
  try { localStorage.removeItem('ee_session_trail') } catch {}
  set({ sessionTrail: [] })
},
```

Do not add `sessionTrail` to the persist middleware partialize — it uses its
own key (`ee_session_trail`) exactly like `sessionWaypoints` uses
`ee_guest_waypoints`. The partialize comment already explains this pattern.

## Constraints

- Only `src/store/mapStore.js` is modified.
- `sessionTrail` must NOT be added to the partialize object (it uses a
  dedicated key outside the Zustand persist bundle).
- `isTracking` is NOT persisted and is NOT changed by this task. When the
  app loads with a non-empty recovered `sessionTrail`, `isTracking` defaults
  to `false`. This is intentional — the user must explicitly resume or discard.
  A future task can add the recover-session UX; this task only saves the data.
- `clearSessionTrail` is also called by `startTracking` (in `useTracks.js`).
  This is correct: starting a new track clears the previous recovery data.
- Write errors (e.g. private mode, quota exceeded) are silently swallowed via
  try/catch — same as the `sessionWaypoints` implementation.

## Definition of Done

- `src/store/mapStore.js` updated as specified.
- An active GPS track that was accumulating points survives a page reload:
  `mapStore.sessionTrail` is non-empty when the app reinitialises.
- Saving or discarding a track (which calls `clearSessionTrail`) also removes
  `ee_session_trail` from localStorage.
- No other files changed.
