# Task 003 — Preserve Learn tab component state across tab switches (V13)

Date: 2026-04-28
Status: PENDING
Source: ux-findings-2026-04-28.md
Branch: main

## Context

V13: When a user navigates away from the Learn tab (e.g., to Map) and returns,
LearnView unmounts and all internal component state is destroyed:
- Current chapter page index in ChapterReader (useState — gone)
- Scroll position in course lists (DOM position — gone)
- Any in-progress navigation within CourseDetail (gone)

Both `free V13` and `guest V13` tests confirmed the state-loss journey.
The architectural cause is in `App.jsx` lines 76–81:

    {activeTab !== 'map' && (
      <div>
        {activeTab === 'dashboard' && <DashboardView />}
        {activeTab === 'settings' && <SettingsView />}
        {activeTab === 'learn' && <LearnView />}
        {activeTab === 'profile' && <ProfileView />}
      </div>
    )}

Conditional rendering (`&&`) causes React to fully unmount the component tree
when `activeTab` changes away from 'learn'. All useState hooks are destroyed.

The fix is CSS-based keep-alive: always render all four non-map tabs, but hide
inactive ones with `display: none`. React keeps the component tree mounted and
all useState values survive. This is the standard mobile-app pattern
(equivalent to UIKit's view hierarchy always being in memory).

IMPORTANT: This is a change to `App.jsx`, which is a shared file. Read AGENTS.md
INTENT protocol before touching it.

### 2. Replace conditional rendering in App.jsx

Current structure (lines 76–85):

    {activeTab !== 'map' && (
      <div style={{ position: 'absolute', inset: 0, zIndex: 0, overflowY: 'auto' }}>
        {activeTab === 'dashboard' && <DashboardView onNavigate={setActiveTab} />}
        {activeTab === 'settings' && <SettingsView onNavigate={setActiveTab} />}
        {activeTab === 'learn' && <LearnView />}
        {activeTab === 'profile' && <ProfileView onNavigate={setActiveTab} />}
      </div>
    )}

Replace with always-mounted tabs, using inline `display` to toggle visibility.
Each tab gets its own wrapper so they do not interfere with each other's scroll
position:

    <div style={{ position: 'absolute', inset: 0, zIndex: activeTab !== 'map' ? 0 : -1, pointerEvents: activeTab !== 'map' ? 'auto' : 'none' }}>
      <div style={{ display: activeTab === 'dashboard' ? 'block' : 'none', height: '100%', overflowY: 'auto' }}>
        <DashboardView onNavigate={setActiveTab} />
      </div>
      <div style={{ display: activeTab === 'settings' ? 'block' : 'none', height: '100%', overflowY: 'auto' }}>
        <SettingsView onNavigate={setActiveTab} />
      </div>
      <div style={{ display: activeTab === 'learn' ? 'block' : 'none', height: '100%', overflowY: 'auto' }}>
        <LearnView />
      </div>
      <div style={{ display: activeTab === 'profile' ? 'block' : 'none', height: '100%', overflowY: 'auto' }}>
        <ProfileView onNavigate={setActiveTab} />
      </div>
    </div>

Key points:
- The outer wrapper uses `zIndex: -1` and `pointerEvents: none` when the map is
  active (same effect as before — non-map content cannot intercept map touch events).
- Each tab has its own scroll container so switching tabs does not reset scroll
  position of other tabs.
- `display: none` keeps React mounted — useState survives — but the DOM subtree
  is not painted or laid out, so there is no performance cost for hidden tabs.
- Remove the `activeTab !== 'map'` conditional wrapper entirely — replace with
  the always-rendered block above.

## Constraints

- BUGS.md rule 32 (DataSheet two-layer architecture) — the non-map tab container
  must not overlap or interfere with the DataSheet z-index (z-index: 20) or
  camera button (z-index: 30). The non-map container should be z-index 0 when
  active and z-index -1 when the map tab is active.
- BUGS.md rule 44 (BottomNav clearance) — do not change padding-bottom on any
  tab's scroll container. The individual view components already handle their
  own bottom clearance.
- Do not add `overflow: hidden` to the outer wrapper — tabs need independent
  scroll containers.
- Do not remove the `overflowY: auto` from individual tab wrappers — LearnView
  and ProfileView are long scrollable pages.
- `onNavigate={setActiveTab}` prop must still be passed to DashboardView,
  SettingsView, and ProfileView — do not drop it.
- Do not wrap LearnView in anything that would block its internal scroll events.
