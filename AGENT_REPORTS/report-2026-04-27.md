## UX Report — 2026-04-27

### Summary
Playwright tests are reporting two significant UX regressions affecting core navigation. The "Dashboard" tab button is obstructed by another UI element, preventing user interaction. Additionally, the "Map" tab button in the bottom navigation is not uniquely identifiable due to other elements in the app sharing similar labels, causing test failures and potential user confusion.

### Findings

1.  **Dashboard Tab Obstruction**
    *   **Description:** The "Dashboard" button in the bottom navigation is covered by an overlapping UI element, making it unclickable. Playwright's `locator.click()` operation timed out after multiple retries, reporting `<div>…</div> from <div>…</div> subtree intercepts pointer events`.
    *   **Likely Cause from Git Diff:** The `src/components/DataSheet.jsx` file received `1 insertion(+)`. Given that the DataSheet is a common UI panel that can dynamically appear and overlay content, it's highly probable that this change, or an interaction it created, has caused the DataSheet to partially obscure the bottom navigation bar upon initial load or after certain actions. This would directly prevent clicks on the "Dashboard" button.
    *   **Severity:** High (Core navigation is blocked, severely impacting app usability).

2.  **Map Tab Selector Ambiguity**
    *   **Description:** The Playwright test fails to uniquely identify the "Map" button in the bottom navigation. It reports a `strict mode violation` because `getByRole('button', { name: 'Map' })` resolves to four elements, including buttons like "My Waypoints", "My Sessions", and "Offline Maps", which are likely located within a dynamic panel like the DataSheet. This indicates a lack of unique accessible naming for the primary "Map" navigation button.
    *   **Likely Cause from Git Diff:** Similar to the Dashboard issue, the `1 insertion(+)` in `src/components/DataSheet.jsx` might have introduced new buttons or re-labeled existing ones within the DataSheet or related panels. The listed ambiguous buttons ("My Waypoints", "My Sessions", "Offline Maps") suggest new features or content within a panel that uses "Map" in its context, leading to collision with the main navigation "Map" button's accessible name.
    *   **Severity:** Medium (Test fragility and potential user confusion if multiple "Map" buttons appear in close proximity without clear distinction. Could be High if users frequently encounter the ambiguity).

### Actionable Prompts for Agents

1.  **[ui] Investigate DataSheet overlay on bottom navigation:**
    Examine `src/components/DataSheet.jsx` and its related CSS/styles. Identify the recent `1 insertion(+)` and determine if it introduced a new element, changed the default `translateY` state, or altered the `z-index` or layout flow such that the DataSheet (or any of its child elements) now obstructs the bottom navigation bar (specifically the "Dashboard" button) on initial render or after a standard interaction. Adjust styling, initial state, or z-index to ensure the bottom navigation is always fully visible and clickable.
2.  **[ui] Resolve 'Map' button naming collision:**
    Review `src/components/DataSheet.jsx` and any components it renders for buttons that include "Map" in their accessible name (e.g., `aria-label`, `name`, or visible text). Ensure the primary "Map" button in the bottom navigation (`aria-label="Map"`) is uniquely identifiable. If other buttons must contain "Map" in their label, refine their accessible names (e.g., "Offline Map Settings", "My Map Waypoints") to prevent collision with the main navigation. Update the Playwright selector in `tests/ux/app.spec.js` to use `getByRole('button', { name: 'Map', exact: true })` for the bottom navigation button once uniqueness is confirmed.

The obstruction of the Dashboard button could be exacerbated or caused by improper handling of these areas on certain devices.

Ensure that button labels are distinct, clear, and unambiguous for users, especially on smaller mobile screens where context can be limited.
