## UX Report — 2026-04-24

### Summary
The Playwright audit revealed that while core application loading and navigation (excluding the map) passed, the critical `map.spec.js` suite was not executed, creating a significant blind spot. Recent changes to `MapLibre paint objects` and the `BottomNav` element, aimed at fixing existing issues, pose a high risk for visual regressions, especially regarding map theming and mobile viewport responsiveness.

### Findings

1.  **Critical Map Test Suite Not Executed**
    *   **Description:** The Playwright results report `stats.expected: 4` tests, all from `app.spec.js`, with no overall errors. However, the recent commit `7ac3b44` explicitly upgraded `tests/ux/map.spec.js` to include critical checks like "assert canvas size" and "paint-parse error check." This `map.spec.js` suite is entirely absent from the report, indicating it was not executed during this pipeline run. This creates a critical blind spot for map-related UX regressions, especially after the recent high-risk changes to `MapLibre paint objects`.
    *   **Likely Cause:** A misconfiguration in the Playwright test runner (`playwright.config.js`) or the CI/CD workflow (`.github/workflows/ux-audit.yml`) preventing `map.spec.js` from being included in the test execution, despite recent efforts to create a "full tests/ux/ spec suite."
    *   **Severity:** High

2.  **Map Theming/Dynamic Styling Regression Potential**
    *   **Description:** Commit `360a79e` involved replacing CSS variables in `MapLibre paint objects`. The project memory (`CLAUDE.md`) explicitly mentions `userStore.theme`, indicating support for dynamic, theme-dependent styling (e.g., light/dark mode). If these CSS variables were previously used to adapt map layer colors, opacities, or other visual properties to the user's selected theme, their replacement with static values will lead to broken or inconsistent map styling that does not respect the user's preferences.
    *   **Likely Cause:** Commit `360a79e [impl] fix BottomNav nav element + replace CSS vars in MapLibre paint objects`. The replacement of CSS variables may have inadvertently hardcoded styling values, ignoring the `userStore.theme` state.
    *   **Severity:** Medium

3.  **Bottom Navigation Visual Regression (Mobile Viewports)**
    *   **Description:** The commit `360a79e` included a "fix BottomNav nav element," which suggests prior styling issues. While `app.spec.js` confirms the navigation is visible, it does not assert its visual correctness or responsiveness. Specifically, the "centre-raised MapPin tab" feature and overall padding/alignment across various mobile viewports (as detailed in `CLAUDE.md`) are at risk. The "fix" might have introduced new visual inconsistencies or layout problems, particularly on mobile devices.
    *   **Likely Cause:** Commit `360a79e [impl] fix BottomNav nav element + replace CSS vars in MapLibre paint objects`. Changes made to address a previous `BottomNav` issue might have inadvertently introduced new layout or styling problems.
    *   **Severity:** Medium
