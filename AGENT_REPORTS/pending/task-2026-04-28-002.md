# Task 002 — Persist guest waypoints to localStorage (V11)

Date: 2026-04-28
Status: PENDING
Source: ux-findings-2026-04-28.md
Branch: main

## Context

V11: Guest users pin waypoints during a session (via WaypointSheet or map tap),
but those pins live only in `mapStore.sessionWaypoints` (in-memory Zustand array).
On reload or app close they are silently destroyed.

The `guest V11` test confirmed the full journey: waypoints added, reload triggered,
waypoints gone. STATE_MAP.md explicitly documents: "Guest waypoints are memory-only
regardless."

Business impact: A guest user who explores the map, drops several interesting
waypoints, and then signs up finds all their pins gone. This breaks the core
prospecting workflow at the exact conversion moment.

`sessionWaypoints` structure (from mapStore.js line 62–64):
```js
sessionWaypoints: [],
addSessionWaypoint: (wp) =>
  set((state) => ({ sessionWaypoints: [...state.sessionWaypoints, wp] })),
```

Each waypoint object has at minimum: `{ id, lat, lng, name, notes, createdAt }`.
Check WaypointSheet.jsx for the exact shape before implementing.

The correct fix at this scope is localStorage persistence for guest waypoints,
with hydration on load. A sign-up migration path (moving localStorage waypoints
into Supabase on first auth) is a separate, larger task — do NOT implement that
here.

## Task

### 1. Persist sessionWaypoints to localStorage

In `mapStore.js`, add a manual localStorage read/write pattern rather than using
`persist` middleware (the store is large with many volatile fields and `persist`
was already applied separately in task-001 with a `partialize` exclusion for
sessionWaypoints). A separate key is cleaner here.

localStorage key: `ee_guest_waypoints`

On store initialisation, read existing waypoints:
```js
const savedWaypoints = (() => {
  try {
    return JSON.parse(localStorage.getItem('ee_guest_waypoints') || '[]');
  } catch { return []; }
})();
```

Initial state:
```js
sessionWaypoints: savedWaypoints,
```

In `addSessionWaypoint`, write back after every add:
```js
addSessionWaypoint: (wp) =>
  set((state) => {
    const next = [...state.sessionWaypoints, wp];
    try { localStorage.setItem('ee_guest_waypoints', JSON.stringify(next)); } catch {}
    return { sessionWaypoints: next };
  }),
```

Add a `clearGuestWaypoints` action (for when the user signs in and their
waypoints are no longer "guest"):
```js
clearGuestWaypoints: () => {
  try { localStorage.removeItem('ee_guest_waypoints'); } catch {}
  set({ sessionWaypoints: [] });
},
```

### 2. Clear guest waypoints on successful sign-in

In the auth success path (wherever `useUserStore` or `useAuthStore` sets the
authenticated user), call `useMapStore.getState().clearGuestWaypoints()` after
a confirmed login so stale guest pins don't mix with the user's saved Supabase
waypoints. Find the sign-in success handler — likely in `useAuth.js` or
`AuthModal.jsx` — and add the call there.

## Constraints

- BUGS.md rule 12/31: do not touch Map.jsx basemap or layer logic.
- BUGS.md rule 36: `showWaypoints` toggle and `saved-waypoints-circles` layer
  are separate from `sessionWaypoints` — do not conflate them.
- Do NOT use `persist` middleware for sessionWaypoints — task-001 already wraps
  mapStore in `persist` with a `partialize` that excludes sessionWaypoints.
  Adding a manual localStorage write inside the action is the right pattern here.
- Guest waypoint objects must be JSON-serialisable — no functions, no class instances.
- Wrap all localStorage calls in try/catch (private mode browsers throw on setItem).
- Do not implement Supabase migration on sign-up — that is out of scope.

## Definition of Done

- `node --check` passes on all modified files.
- `npm run build` exits 0.
- On local dev as guest: add a waypoint, hard-reload → waypoint is still visible on map.
- On local dev: sign in → guest waypoints are cleared from the map and from
  `ee_guest_waypoints` in localStorage.
- `localStorage.getItem('ee_guest_waypoints')` returns `null` after sign-in.
