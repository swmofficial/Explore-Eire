## Context

V7 (theme resets on reload) is confirmed by a **hard test assertion**, not
screenshot inference:

```js
// guest V7 test:
if (flipped) {
  expect(tReloaded).toBe('dark');   // PASSES → theme genuinely reset
}
```

This test can only pass if the theme is 'dark' after reload. If the persist
middleware were hydrating correctly, `tReloaded` would be 'light', and the
assertion would FAIL. It passes consistently across guest and free tiers,
across two pipeline runs (14:18 and 15:35 on 2026-04-28), with the 180s
Vercel wait in place. The code is on the deployed server.

The Zustand `persist` middleware IS configured correctly in `userStore.js`
(name: 'ee-user-prefs', partialize includes theme, isPro, subscriptionStatus).
Yet theme is not surviving reload. The root cause is empirically confirmed but
architecturally opaque — the middleware writes to localStorage correctly but
the hydration on the subsequent load is not reflected in the `data-theme`
attribute when the test reads it.

**Proven alternative**: The manual IIFE + write pattern used for
`sessionWaypoints` (task-002, `ee_guest_waypoints`) and `sessionTrail`
(task-006, `ee_session_trail`) works reliably in this environment. Both use:
1. IIFE on store creation to read from localStorage
2. Explicit `localStorage.setItem` in the action that changes the value
3. No dependency on Zustand persist middleware for that field

This task applies the same proven pattern to `theme`.

## Task

Modify `src/store/userStore.js` only. Three changes:

### 1. Remove `theme` from the persist partialize

```js
// BEFORE:
partialize: (state) => ({
  theme: state.theme,
  isPro: state.isPro,
  subscriptionStatus: state.subscriptionStatus,
}),

// AFTER:
partialize: (state) => ({
  isPro: state.isPro,
  subscriptionStatus: state.subscriptionStatus,
}),
```

`ee-user-prefs` will no longer contain a `theme` key. The persist
middleware continues to handle `isPro` and `subscriptionStatus` normally.

### 2. Initialise `theme` from localStorage via IIFE

```js
// BEFORE:
// Colour theme — 'dark' | 'light'
theme: 'dark',

// AFTER:
// Colour theme — 'dark' | 'light'
// Persisted via ee_theme (manual pattern — Zustand persist not reliable
// for this field in the deployed environment; see task-008 notes).
theme: (() => {
  try { return localStorage.getItem('ee_theme') || 'dark' } catch { return 'dark' }
})(),
```

### 3. Write to localStorage in setTheme

```js
// BEFORE:
setTheme: (t) => set({ theme: t }),

// AFTER:
setTheme: (t) => {
  try { localStorage.setItem('ee_theme', t) } catch {}
  set({ theme: t })
},
```

## Constraints

- Only `src/store/userStore.js` is modified.
- `isPro` and `subscriptionStatus` remain in the persist partialize — do not
  touch them. The reliability issue has only been confirmed for `theme` via
  direct test assertion. The other two fields have no direct assertion evidence
  of failure; switching them prematurely would be speculative.
- The store's public API is unchanged: `theme` and `setTheme` remain at the
  same keys. No other file needs updating.
- The `ee_theme` key joins `ee_guest_waypoints` and `ee_session_trail` as a
  manually-managed localStorage key. It is never cleared (theme has no "sign-out
  reset" requirement — dark mode preference belongs to the device, not the account).
- Write errors (private mode, quota) are silently swallowed via try/catch.
- After this change, `ee-user-prefs` in localStorage will still contain
  `isPro` and `subscriptionStatus` from existing persist middleware. Any prior
  `theme` key in `ee-user-prefs` is harmless — persist middleware will just not
  use it on hydration.
