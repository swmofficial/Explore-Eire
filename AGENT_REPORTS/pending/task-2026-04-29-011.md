# Task 011 — P1 Pro badge race: fix global-setup storageState timing

Date: 2026-04-29
Status: PENDING
Source: ux-findings-2026-04-29.md (finding 3)
Branch: main

## Context

`pro P1` annotation: `pro-badge-count: 8 (expected: 0 for Pro tier)`. The LayerPanel
conditional `{layer.pro && !isPro && <ProBadge />}` is CORRECT (LayerPanel.jsx:114).
The bug is that `isPro` is `false` when the badge count is read.

Two-layer race:

**Layer 1 — storageState captured before isPro is persisted:**
`global-setup.js` logs in the pro test account, waits 1000ms, then calls
`context.storageState()`. The `useSubscription` hook fetches `profiles.is_pro` from
Supabase and calls `setIsPro(true)`, which triggers the Zustand persist middleware to
write `{ state: { isPro: true, subscriptionStatus: 'active' }, version: N }` to
`ee-user-prefs` in localStorage. If 1000ms is not enough for the Supabase round-trip to
complete, the captured `ee-user-prefs` in `pro.json` has `isPro: false`. Every test that
loads with this storageState starts with `isPro: false` in the persisted store.

**Layer 2 — test reads badge count before async profile fetch completes:**
Even when the storageState has `isPro: true`, the test's `waitForTimeout(2500 + 2000)`
might not be enough for the profile fetch in CI. task-009 added a 2000ms wait but the
annotation still shows 8 badges.

## Task

### Part A — global-setup.js

Replace the `await page.waitForTimeout(1000)` before `context.storageState()` with a
poll that waits for `ee-user-prefs` to contain `isPro: true`:

```js
// After login, wait until isPro is persisted (Supabase profile fetch may be slow in CI)
const isPersisted = await page.waitForFunction(
  () => {
    try {
      const s = JSON.parse(localStorage.getItem('ee-user-prefs') || '{}');
      return s?.state?.isPro === true;
    } catch { return false; }
  },
  { timeout: 15000 }
).catch(() => false);

if (!isPersisted) {
  console.warn('[global-setup] isPro not persisted within 15s — storageState may have isPro:false');
}
```

Remove the `await page.waitForTimeout(1000)` line that follows the modal close wait.

### Part B — tests/ux/pro.spec.js (P1 test)

In the `pro P1` test, replace the two `page.waitForTimeout` calls before reading badge
count with a `waitForFunction` that waits until `isPro: true` is in localStorage. Fall
back to 10s timeout:

```js
// Wait for isPro to be set (Supabase profile fetch must complete before we read badges)
await page.waitForFunction(
  () => {
    try {
      const s = JSON.parse(localStorage.getItem('ee-user-prefs') || '{}');
      return s?.state?.isPro === true;
    } catch { return false; }
  },
  { timeout: 10000 }
).catch(() => {});
```

Replace the existing:
```js
await page.waitForTimeout(2500);
await page.waitForTimeout(2000);
```
with the `waitForFunction` above. Keep the `await page.locator('#tour-layers-btn').first().waitFor(...)` that follows it unchanged.

## Constraints

- `global-setup.js` is not a shared app source file — no INTENT block required.
- `pro.spec.js` is not a shared app source file — no INTENT block required.
- Do NOT change any application source files.
- The `waitForFunction` must handle `localStorage.getItem` returning null gracefully
  (the `|| '{}'` fallback handles this).
- The `catch(() => false)` / `catch(() => {})` ensures the test never hard-fails if
  isPro takes > timeout to appear.

## Definition of Done

- `global-setup.js` polls for `isPro: true` in localStorage before saving storageState.
- `pro.spec.js` P1 test uses `waitForFunction` instead of fixed timeouts.
- On next pipeline run: `pro P1` annotation shows `pro-badge-count: 0` (or very low)
  and `upgrade-sheet-shown: no`, confirming the race is resolved.
