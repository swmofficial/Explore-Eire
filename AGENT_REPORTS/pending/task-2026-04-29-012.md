# Task 012 — V7 theme resets: persist version bump + test assertion fix

Date: 2026-04-29
Status: PENDING
Source: ux-findings-2026-04-29.md (finding 5)
Branch: main

## Context

`guest V7` and `free V7` both show `theme-after-reload: dark` on this run, even though
task-008 (commit 31c0988) implemented the manual `ee_theme` localStorage pattern in
`userStore.js`. The STATUS.md Implementer note stated the V7 test should now FAIL
(assertion `expect(tReloaded).toBe('dark')` would fail when theme persists). Instead the
test PASSES, meaning `tReloaded` is still 'dark'. Task-008 is not working.

**Root cause — stale `ee-user-prefs` stored state:**

Before task-008, `userStore`'s persist config had `theme` in the `partialize`. The stored
`ee-user-prefs` JSON was: `{ state: { isPro: false, subscriptionStatus: 'free', theme: 'dark' }, version: 0 }`.

After task-008, `partialize` no longer includes `theme`. But Zustand's persist middleware
still reads the OLD stored JSON on hydration and merges it with current state. The merge
strategy is `{ ...currentState, ...persistedState }` — persistedState wins. So the old
`theme: 'dark'` key in the stored JSON overwrites the IIFE-set `theme: 'light'` value.

This affects:
- **Free/Pro users** — `pro.json`/`free.json` storageState was captured while `theme` was
  in the partialize, so it contains `theme: 'dark'` in `ee-user-prefs`.
- **Guest users** — even a fresh context can accumulate `ee-user-prefs` entries from the
  previous version if the browser has ever run the old code (pre-task-008). In CI this
  means the store file is cached or the storageState from a pre-task-008 run is still in
  `.auth/`.

The manual IIFE in task-008 reads `ee_theme` correctly (it's the right key and the write
is correct). The problem is Zustand's hydration calling `setState({ ..., theme: 'dark' })`
AFTER the IIFE has set `theme: 'light'`, overwriting it.

**Fix: persist version bump + migration**

Zustand persist supports `version` and `migrate`. Incrementing version from 0 to 1 and
providing a migration that strips `theme` from stored state prevents the overwrite.

## Task

### Part A — src/store/userStore.js (INTENT REQUIRED — see below)

In the `persist(...)` config object, add `version: 1` and `migrate`:

```js
{
  name: 'ee-user-prefs',
  version: 1,
  migrate: (persistedState, version) => {
    if (version < 1) {
      // Remove stale theme field written by pre-task-008 code
      const { theme: _theme, ...rest } = persistedState
      return rest
    }
    return persistedState
  },
  partialize: (state) => ({
    isPro: state.isPro,
    subscriptionStatus: state.subscriptionStatus,
  }),
},
```

The `migrate` function runs when Zustand reads a stored state with a version < current.
After migration, `theme` is absent from the merged state, so the IIFE value ('light' or
whatever was in `ee_theme`) survives hydration.
