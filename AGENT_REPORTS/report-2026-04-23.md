## UX Report — 2026-04-23

### Summary
A critical regression has been introduced by the recent infrastructure commit, causing all Playwright UI tests to fail at the initial navigation step. This is due to a misconfiguration of the `baseURL` in `playwright.config.js`, stemming from an environment variable naming mismatch and potentially improper handling within the CI environment. All further UI testing is currently blocked.

### Findings
1.  **Critical Failure: All Playwright UI tests blocked due to invalid `baseURL`.**
    *   **Description:** The Playwright test `app loads and shows splash screen` fails immediately with the error `Error: page.goto: Protocol error (Page.navigate): Cannot navigate to invalid URL` when attempting to navigate to the root path (`/`). This indicates that the `baseURL` configured for Playwright is not resolving to a valid URL string. As this is the very first step in the test suite, all subsequent UI tests are prevented from running.
    *   **Likely Cause from Git Diff:**
        *   **Environment Variable Mismatch:** The `playwright.config.js` was updated in the recent commit (visible in `git diff`) to expect `process.env.VITE_APP_URL` for its `baseURL`. However, the `PROJECT MEMORY` indicates that the standard application URL environment variable for Vercel deployments and general use is `APP_URL`. This mismatch means `process.env.VITE_APP_URL` is likely `undefined` or an empty string in the CI environment.
        *   **CI Environment Handling & Fallback Issue:** Although the updated `playwright.config.js` includes a fallback (`|| 'http://localhost:5173'`), the specific error `Cannot navigate to invalid URL` implies that the effective `baseURL` at runtime was still an invalid string (e.g., `""`), overriding or otherwise bypassing the intended fallback to `http://localhost:5173`. This could stem from `VITE_APP_URL` being explicitly set to an empty string in CI, or a subtle interaction with `dotenv.config()` preventing the fallback from correctly applying.
    *   **Severity:** High (Regression, completely blocks automated UI testing).

### Actionable Prompts for Agents
1.  **[ui] Update Playwright `baseURL` configuration:** Modify `playwright.config.js` to correctly reference the `APP_URL` environment variable (e.g., `baseURL: process.env.APP_URL || 'http://localhost:5173'`) to align with existing project infrastructure and Vercel environment variables.
2.  **[infra] Ensure `APP_URL` is correctly passed to Playwright tests in CI:** Review the `.github/workflows/ux-test.yml` workflow. Verify that the `APP_URL` environment variable (containing the full application URL for the deployed environment, e.g., `https://explore-eire.vercel.app`) is explicitly defined and passed as an environment variable to the Playwright test runner.
3.  **[infra] Investigate Playwright `baseURL` fallback failure:** If the `Cannot navigate to invalid URL` error persists after addressing the variable naming and CI passing, debug the CI environment to determine the exact value of `process.env.APP_URL` (or `process.env.VITE_APP_URL` if that naming is kept) and the resulting `baseURL` within the Playwright execution context. The fallback `|| 'http://localhost:5173'` should prevent an invalid `baseURL` string.

### Flagged for Human Review
*   **Environment Variable Strategy Consistency:** The recent commit's subject mentions `BASE_URL direct from secret`, while `playwright.config.js` changed from `BASE_URL` to `VITE_APP_URL`, and the project memory highlights `APP_URL`. A consistent and clearly documented strategy for managing application URLs (including `BASE_URL`, `APP_URL`, `VITE_APP_URL`) across local development, CI, and production deployments, as well as how secrets are integrated, should be established to avoid future misconfigurations and confusion.
*   **Impact of `dotenv.config()` in CI:** The addition of `dotenv.config()` to `playwright.config.js` should be reviewed for its behavior in the CI environment. While beneficial for local development, it might interact unexpectedly with environment variables already set by the CI pipeline, especially if a `.env` file exists but is empty in the CI context.
